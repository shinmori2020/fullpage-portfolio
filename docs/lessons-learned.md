# スナップスクロールサイト 反省点と次回への指針

## 1. 今回のプロジェクトで発生した問題

### 1-1. レスポンシブ対応に想定以上の工数がかかった

- PCでは2〜3カラムで情報を横に並べていたが、SPでは100vh固定のため縦に積むとコンテンツが溢れた
- 27枚のスライド全てに個別のレスポンシブ対応が必要になった
- ブレイクポイントの変更（900px→1024px/768px/480px）で既存CSSの書き直しが発生した

### 1-2. position: absoluteの多用による技術的負債

- PC表示のために`position: absolute`で自由に配置した要素が、レスポンシブで`position: relative`に戻す際に毎回競合した
- `left/top/right/bottom/transform/z-index`を全てリセットする必要があり、`!important`の乱用につながった
- 例: スライド5の装飾帯、スライド6の全要素、スライド10のブルー帯と写真

### 1-3. 共通クラスの変更による副作用

- `.placeholder-image`の`position: static`化が全スライドに影響し、スライド1と27のプレースホルダーが崩れた
- 共通クラスのレスポンシブ変更は慎重に行う必要がある

### 1-4. 情報量の多いスライドが多すぎた

- 1セクションに複数の情報を詰め込んだスライドほどレスポンシブが困難だった
- 例: スライド6（カード3枚+写真+名前+説明+肩書き）、スライド19（3カラム+4アイコン+写真）

### 1-5. デザインの参照元が不明確だった

- slide-layouts.mdの文章定義と実際の参考画像（pages.jpg）が一致しない箇所があった
- 小さい参考画像から詳細を読み取るのが困難で、添付画像を最優先する方針に途中で変更した

### 1-6. レイアウトパターンが多すぎた

- 27枚で11種類以上のレイアウトパターンを使用した
- パターンごとにレスポンシブのアプローチが異なり、作業量が膨らんだ

## 2. 技術的な教訓

### 2-1. CSSの設計

| やるべきだったこと | 今回やったこと |
|---|---|
| Flexbox/Gridベースでレイアウト | position: absoluteを多用 |
| モバイルファーストで記述 | PCファーストで記述 |
| レスポンシブCSSを最初から分離 | 後から分離（移行作業が発生） |
| 共通クラスは変更しない | 共通クラスを変更して副作用 |
| rem/vw/vhで統一 | px固定値を多用 |

### 2-2. HTMLの設計

| やるべきだったこと | 今回やったこと |
|---|---|
| 装飾要素をlayout内に配置 | layout外に配置してorder不能 |
| セマンティックな構造 | スライドごとに構造がバラバラ |
| クラス名の統一規則 | スライド9/15でクラス共有→後で分離 |

### 2-3. レスポンシブの設計

| やるべきだったこと | 今回やったこと |
|---|---|
| 3段階のブレイクポイントを最初から決定 | 途中で900px→1024px/768px/480pxに変更 |
| 各スライドのSP表示を事前にデザイン | PCデザインのみで即実装 |
| zoom/scale方式を最初に検証 | 途中で試して不採用 |

## 3. 次回のスナップスクロールサイト制作指針

### 3-1. デザイン段階

1. **PC/SP両方のカンプを用意する** — PCだけのデザインからSPを推測しない
2. **1セクション = 1メッセージ** — 情報量を「タイトル + テキスト + 要素1つ」に制限
3. **レイアウトパターンを4種類以内に絞る**

### 3-2. 推奨レイアウトパターン

**A: 中央テキスト型**（Cover, Thanks, パート区切り向け）
```
┌─────────────────┐
│                 │
│   タイトル       │
│   テキスト       │
│                 │
└─────────────────┘
```
- レスポンシブ: 変更不要
- 使用例: スライド1, 27

**B: 左右2分割型**（メインコンテンツ向け）
```
PC:  [画像 | テキスト]
SP:  [画像]
     [テキスト]
```
- レスポンシブ: flex-direction変更のみ
- 使用例: スライド4, 13

**C: 上下2分割型**（リスト/カード向け）
```
PC:  [見出し+説明]
     [カード3枚横並び]
SP:  [見出し]
     [カード縦並び]
```
- レスポンシブ: カードの並び方向変更のみ
- 使用例: スライド12

**D: フルビジュアル型**（インパクト向け）
```
PC/SP: [背景画像+オーバーレイ]
       [テキスト前面]
```
- レスポンシブ: 変更不要
- 使用例: スライド16

### 3-3. 避けるべきパターン

- 3カラム以上のレイアウト（SPで収まらない）
- position: absoluteによる重ね配置（レスポンシブでの位置リセットが困難）
- 画面端に接する装飾要素（ブレイクポイントごとに位置調整が必要）
- 複数の写真+複数のテキストブロックの組み合わせ

### 3-4. 推奨セクション構成（27枚の場合）

| 枚数 | 用途 | レイアウト |
|---|---|---|
| 2枚 | Cover / Thanks | A: 中央テキスト型 |
| 4枚 | パート区切り | A: 中央テキスト型 |
| 8枚 | メインコンテンツ | B: 左右2分割型 |
| 4枚 | リスト/カード | C: 上下2分割型 |
| 4枚 | ビジュアル | D: フルビジュアル型 |
| 5枚 | データ/チャート | B or C |

### 3-5. CSS設計方針

1. **モバイルファーストで記述** — `min-width`で拡張する
2. **Flexbox/Gridのみ使用** — `position: absolute`は装飾のみ
3. **レスポンシブCSSは最初からファイル分離**
4. **共通クラスのレスポンシブは触らない** — スライド固有セレクタで上書き
5. **単位はrem/vw/vhを基本** — px固定値は最小限
6. **ブレイクポイントは最初に確定** — 1024px / 768px / 480px

### 3-6. 実装フロー

1. HTMLの骨格を全スライド分作成
2. モバイル用CSS（1カラム）を先に記述
3. タブレット用拡張（`min-width: 768px`）
4. PC用拡張（`min-width: 1024px`）
5. 各ブレイクポイントで全スライドを確認
6. 微調整
